# Case Study Method

## Overview

The Case Study Layer applies the Synchronization Audit Protocol to concrete examples.

v0.1 defined the audit ruler.

v0.2 uses that ruler.

The purpose of this layer is not to accuse, prove theft, or overstate influence. The purpose is to record observed similarities in a conservative, repeatable, evidence-based format.

## Principle

A case study must not begin with a conclusion.

It must begin with an observed similarity.

The case study then asks:

1. What is the source origin?
2. What is the external target?
3. What appears similar?
4. What appears different?
5. What evidence exists?
6. Which A–E classification is most conservative?
7. What additional evidence would be required to strengthen or revise the classification?

## Classification Discipline

The protocol uses five fixed classifications:

| Code | Label | Meaning |
|---|---|---|
| A | `accidental_sync` | Similarity appears coincidental or too weak to support a stronger claim. |
| B | `structural_convergence` | Similarity is likely explained by shared constraints or environmental pressure. |
| C | `indirect_influence` | Influence is plausible, but no direct reference is proven. |
| D | `direct_reference_or_imitation` | Direct reference, imitation, citation, or derivation is supported by evidence. |
| E | `ai_reconstruction` | Similarity may have been reconstructed, amplified, or reassembled by an AI system. |

## Conservative Default

When no direct reference path exists, do not assign D.

When independent physical, economic, social, technical, or historical constraints can explain the similarity, prefer B.

When public accessibility exists but no direct citation exists, C may be considered, but only if the similarity is stronger than general convergence.

When AI context, retrieval, memory, or prompt dependency may have reconstructed the similarity, consider E.

## Case Study Format

Each case study should include:

```text
1. Summary
2. Source Origin
3. External Target
4. Observed Similarity
5. Evidence Matrix Review
6. Classification
7. Conservative Conclusion
8. Recommended Action
What Case Studies Must Avoid

Case studies should avoid:

claiming direct influence without trace evidence
treating similarity as proof
using emotionally loaded accusation language
collapsing AI reconstruction into theft
treating public accessibility as proof of use
treating vocabulary overlap as strong evidence by itself
Stronger Evidence

The following evidence is stronger than vocabulary overlap:

matching structural order
matching unusual metaphor
documented reference path
copied diagram or sequence
commit history relationship
citation or backlink
prompt or retrieval evidence
AI output that reproduces distinctive structure from known context
Weaker Evidence

The following evidence is weaker:

broad shared terms
common industry vocabulary
similar timing alone
public accessibility alone
generic conceptual overlap
similarity caused by obvious external constraints
v0.2 Scope

v0.2 adds case-study documentation and multiple example records.

It does not yet define a structural fingerprint algorithm.

That is reserved for v0.3.

Final Rule

Case studies are not verdicts.

They are audit records.

The goal is not to win an argument.

The goal is to keep similarity, influence, convergence, reconstruction, and evidence separate.
